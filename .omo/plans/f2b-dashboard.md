# Fail2Ban Live Dashboard Web App

## TL;DR

> **Quick Summary**: Build a lightweight Fail2Ban attack intelligence dashboard using Bash scripts + Cron + static vanilla JS frontend. Parse local fail2ban.log files into aggregated JSON, display 9 visualization sections with Chinese/English i18n, dark/light themes, and configurable auto-refresh.
>
> **Deliverables**:
> - Bash log parser script (`f2b-parse.sh`) with aggregation engine
> - Bash GeoIP lookup script (`f2b-geoip.sh`) with IP-API + local cache
> - Static single-page dashboard (HTML/CSS/JS) with ECharts (CDN)
> - i18n system (中文/English) with JSON translation files
> - Dark/Light theme system with CSS variables + localStorage persistence
> - Install script (`install.sh`) with cron setup
> - Config file (`config.js`) for customizable settings
>
> **Estimated Effort**: Medium
> **Parallel Execution**: YES - 4 waves
> **Critical Path**: JSON Schema → Bash Parser → Frontend Components → Integration → Install

## Context

### Original Request
開發一個 Fail2Ban Live Dashboard Web App，用來顯示即時系統遭受攻擊的情資，資料來源是分析 Fail2Ban 的本地日誌檔。網頁的所有數據要能定時刷新，可選擇不同的時間間隔。App 部署要簡單，相容所有 Linux 發行版，系統依賴性低。中文+英文介面，響應式設計（移動裝置優先），單頁式設計，深色+淺色主題，高級感，極簡風格。

### Interview Summary
**Key Discussions**:
- Architecture: Bash scripts + Cron + static vanilla JS frontend — NO backend server
- Frontend: Pure HTML/CSS/JS with ECharts from CDN — NO npm, NO build step
- GeoIP: IP-API online lookup with local geo-cache.json
- Auth: Not needed (single machine / internal network)
- Static serving: Not included — user configures their own web server
- Refresh mechanism: Cron every 5 min updates JSON, frontend JS polls at configurable interval
- Dashboard: All 9 chart types + per-jail service reports

**Research Findings**:
- 3 log files: .1 (4041 lines, Jun 17-19), .2 (22675 lines, Jun 9-16), .3 (31193 lines, Jun 1-8)
- Total: ~57,909 lines, ~19 days, ~3000 events/day
- Event types: Found, Ban, Unban, Restore Ban, Ignore, "already banned", "Determined IP using DNS Lookup"
- 2 jails: sshd (banTime=7200s, findtime=900s, maxRetry=4), asterisk (banTime=259200s, findtime=600s, maxRetry=4)
- ~200+ unique IPs, top attacker: 165.227.124.92 (2802 occurrences)
- IPv6 addresses present; private IPs present in asterisk logs
- IP-API free tier: 45 req/min, HTTP only, no IPv6 geo, no batch on free tier

### Metis Review
**Identified Gaps** (all addressed):
- JSON must contain AGGREGATED data, not raw events → critical for browser performance
- Missing current fail2ban.log handling → parser must handle both current + rotated files
- Found event deduplication → count per occurrence, no dedup
- Restore Ban handling → separate category, not "new bans"
- Flush ticket filtering → exclude from unban statistics
- IPv6 GeoIP → label as "IPv6 — Geo unavailable"
- Private IPs → skip geo-lookup, label as "Internal/Private"
- Atomic JSON writes → write to .tmp then mv
- Cron lock → use flock to prevent concurrent runs
- .gz rotated logs → skip initially (document as limitation)
